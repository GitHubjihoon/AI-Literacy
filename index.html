Module 1

쓰레드 & 세션

독립된 연산 공간, 그리고 세션 오염

개념 — 쓰레드는 '독립된 연산 공간'

대화창(쓰레드) 하나 = AI가 그 안의 내용만 기억하는 독립된 작업실입니다.

옆 대화창의 내용은 넘어오지 않습니다.

세션 오염(Context Pollution)

한 창에서 여러 업무를 섞으면, 앞 업무의 잔재가 뒤 답변의 품질을 떨어뜨립니다.

"Each Claude Code session begins with a fresh context window."
— 모든 Claude Code 세션은 새 컨텍스트 윈도우(독립 작업 공간)에서 시작합니다. 출처: Claude Code 공식 문서 — How Claude remembers your project
M1 심화 · 세션 오염

한 창에 여러 업무를 섞으면? — 컨텍스트가 오염된다

같은 대화창에서 전혀 다른 주제를 이어 물으면, 앞 업무의 잔재가 남아 AI의 '주의'가 흐려집니다.
→ 클릭(→)으로 서로 다른 업무를 한 창에 섞어 보세요.

🧵 하나의 대화창에 계속 이어 묻기
① 분기 매출 보고서 작성 ② Python 버그 수정 요청 ③ 인스타그램 마케팅 카피 ④ 경쟁사 시장 분석
맥락 선명도 (답변 품질)
100%
AI의 '현재 주의' 상태
맑음 ✨
✅ 해결 — 업무가 바뀌면 새 쓰레드로 분리하세요. 보고서·코드·카피를 각자 깨끗한 창에서 다루면 선명도 100%가 유지됩니다. 출처: Claude Code 공식 문서 — "Each Claude Code session begins with a fresh context window."

실무 의미 — 대화창을 언제 갈아야 할까

  • 업무가 바뀌면 새 창 — 보고서 작성 → 코드 리뷰처럼 맥락이 다르면 분리
  • 답변이 갑자기 엉뚱해지면 오염 신호 → 초기화 타이밍
  • 하나의 목표 = 하나의 쓰레드. "한 창에 다 때려넣기"가 품질 저하의 시작

💡 핵심: 깨끗한 작업실에서 깨끗한 결과가 나온다.

참고독립 컨텍스트 개념은 서브에이전트에서도 동일하게 적용됩니다 — "Each subagent runs in its own context window."
M1 → M2 연결

하나의 쓰레드 = 쌓이는 컨텍스트 = 거대한 세션

대화창을 닫지 않으면 모든 질문·답변·첨부 문서가 한 세션에 계속 누적됩니다.
→ 클릭(또는 →)으로 컨텍스트가 차오르는 과정을 보세요.

컨텍스트 한계 (예: 200K 토큰)
3%
시스템 + CLAUDE.md (기본 적재) ＋ 대화 ① 질문·답변 ＋ 대화 ② 추가 요청 ＋ 문서 첨부 (PDF·CSV) ＋ 대화 ④·⑤ 누적 ＋ 한계 근접 ⚠️ 앞부분 망각 시작
누적된 세션이 한계에 가까워질수록 정확도·회상력이 떨어집니다(context rot).
→ 그래서 ① 업무가 바뀌면 새 쓰레드로 분리하고, ② 긴 문서는 통째로 넣지 말고 RAG(M5)로 필요한 조각만 넣습니다. 다음 장(M2)에서 이 '한계'와 'context rot'를 자세히 다룹니다.

컨텍스트 윈도우

AI의 메모리 한계와 정보 배치

컨텍스트 윈도우 = AI의 '작업 기억'

AI가 한 번에 볼 수 있는 텍스트의 총량이자, 지금 이 대화창이 들고 있는 모든 것. 컴퓨터의 RAM(작업 기억)과 비슷해서 한계를 넘으면 앞부분을 잊습니다.

"The 'context window' refers to all the text a language model can reference when generating a response … it represents a working memory for the model." 출처: Claude 공식 문서 — Context windows

Lost in the Middle = 공식 용어 "Context Rot"

정보가 많을수록 좋은 게 아닙니다. 토큰이 늘수록 정확도·회상력이 떨어지는 현상을 Anthropic은 context rot(컨텍스트 부패) 라고 부릅니다.

"As token count grows, accuracy and recall degrade, a phenomenon known as context rot. This makes curating what's in context just as important as how much space is available." 출처: Claude 공식 문서 — Context windows

Module 3

토큰 (Token)

글자 수가 아닌 연산 단위

AI는 '글자'가 아니라 '토큰'으로 센다

토큰 = AI가 텍스트를 처리하는 최소 연산 단위. 단어·일부 음절 단위로 쪼개집니다.

영어

"artificial" ≈ 2토큰. 효율 높음

한국어

받침·조사로 토큰이 더 많이 쪼개짐 → 같은 의미라도 비용↑

실무 의미 — 비용과 한계의 최적화

  • 토큰 = 비용이자 컨텍스트 한계. 토큰을 아끼면 둘 다 좋아진다
  • 불필요한 인사말·중복 설명 제거 → 한계 도달을 늦춘다
  • 대량 반복 작업은 간결한 프롬프트가 곧 비용 절감

💡 "짧고 명확하게"는 매너가 아니라 엔지니어링이다.

참고토큰 정의와 사전 측정은 공식 문서 참고 — Glossary (Tokens) · Token counting

레이턴시 & 온도

느려짐의 원인과 답변 무작위성 제어

AI는 답을 '한 단계씩' 만들며 도달한다

즉시 튀어나오는 게 아니라, 다음 말을 하나씩 예측하며 추론을 쌓습니다(자기회귀) — 그래서 레이턴시(지연)가 생깁니다.

🧠 생각 중…

하루 단위니까 올림 → 15일째

Module 5 · 핵심 ★

RAG

검색 증강 생성 (Retrieval-Augmented Generation)

사내 문서·지식창고를 AI에 연동할 때 내부에서 벌어지는 일

실무에서 가장 많이 하는 질문

"문서를 분명히 줬는데,
AI가 왜 엉뚱한 소리를 하죠?"

→ AI는 첨부 문서를 처음부터 끝까지 읽지 않습니다.

AI의 지식은 '어디서' 올까?

"외부 지식 = 웹 검색"으로 오해하기 쉽습니다. 출처를 3가지로 나누면 명확해져요.

🧠

모델이 외운 지식

학습된 일반 상식

빠르지만 최신·사내 정보는 모름, 가끔 지어냄(환각).

📄

사내 문서·컨텍스트

RAG ← 이 모듈

우리가 전달한 업로드 문서·사내 지식창고에서 관련 조각을 찾아 주입. 정확·맞춤.

🌐

웹 검색

실시간 외부 사이트

필요할 때 인터넷에서 검색. 공개된 최신 정보용.

💡 흔한 오해 — "외부 지식 = 웹"이 아닙니다.
학술 용어로 '외부'는 모델 바깥(학습에 없던) 지식 전체(📄+🌐)를 뜻합니다. 실무에선 📄 사내 문서(전달) / 🌐 웹(검색) 으로 구분하세요. RAG는 📄 사내 문서를 가져오는 기술입니다. 출처: Anthropic 공식 — Introducing Contextual Retrieval
자주 하는 오해

"AI랑 대화하면 내가 이 AI를 학습시킨다?"

💬 "앞으론 이렇게 해줘"
💬 "내 취향 기억해"
🧠🔒
학습 완료 · 고정
대화는 튕겨 나감 — 모델 안 바뀜 ✕
❌ 오해

대화를 많이 할수록 이 모델이 나에게 맞춰 똑똑해진다.

✅ 사실

모델은 이미 학습이 끝나 고정. 대화는 그 세션에서만 쓰이고, 끝나면 모델은 그대로.

학습 ≠ 대화 ≠ 기억 — 헷갈리지 않기

그럼 AI는 어떻게 '나를 기억'하는 것처럼 보일까요? 셋을 구분하면 명확합니다.

🏭

학습 (Training)

AI 회사가 미리·대량으로 가중치를 조정. 당신이 쓰기 전에 끝남.

💬

대화 (추론)

가중치는 안 바뀜. 그 자리에서 답만 생성하고 끝.

📄

기억

'기억'처럼 보이는 건 저장 파일·RAG 덕분 — 모델이 변하는 게 아님.

연결11번에서 봤듯 대화는 휘발됩니다. 지속은 모델이 아니라 저장 파일(CLAUDE.md)·RAG의 몫이에요.

※ AI 회사가 다음 버전 모델을 만들 때 (정책·동의 하에) 데이터를 쓸 수는 있지만, 지금 이 모델이 실시간으로 바뀌는 것과는 다릅니다. (업무용 플랜은 학습 미사용인 경우가 많음)

RAG는 3단계로 작동한다

STEP 1
조각내기 & 임베딩

문서를 잘게 쪼개(Chunk) 의미 좌표 공간(Vector Store)에 정렬해 둔다.

→
STEP 2
유사도 검색

질문이 들어오면 전체가 아닌, 질문과 가장 비슷한 조각만 뽑는다.

→
STEP 3
맥락 주입 & 생성

뽑은 조각을 프롬프트에 끼워넣고 "이걸 근거로 답해"라고 지시(Generation).

즉, AI는 문서 전체가 아니라 검색된 일부 조각만 보고 답한다.

참고표준 RAG는 임베딩 + BM25(키워드 매칭)를 함께 써 정확도를 높입니다 — Anthropic, Contextual Retrieval

🛠️ "도구를 쥐여준다"의 정확한 의미

LLM은 원래 글자만 출력합니다 — 스스로 파일을 읽거나 검색·실행하지 못해요. '도구를 쥐여준다'는 두 가지를 해준다는 뜻입니다.

① 가능한 행동 목록 알려주기

각 도구를 이름·설명·입력형식으로 모델에게 전달. 예: 사내문서검색(query)

② 실제 실행 코드 연결

모델은 "검색해줘"라고 요청만 하고, 실제 실행은 주변 프로그램이 수행

흐름  모델: "사내문서검색(query=연차) 써줘" → 프로그램: 실제 검색 → 결과 전달 → 모델: "연 15일입니다" (필요하면 반복 = 루프)

💡 결정은 모델, 실행은 코드. 연결한 코드만큼만 할 수 있고, 도구의 설명(description)이 모델의 선택을 좌우합니다. (예: Read · Bash · Grep · WebSearch · MCP)

🔧 도구 = 망치(행동) · 📘 스킬 = 조립 설명서(절차) — 서로 다릅니다.

스킬(Skills) — 반복 작업을 '함수'로 포장

Skills = 절차·도구 모듈

자주 쓰는 지시·체크리스트·절차를 SKILL.md 파일로 저장. (예: 사내 보고서 양식, 규정 검색)

프롬프트의 함수화

잘 만든 프롬프트에 이름을 붙여 저장 → 입력만 바꿔 언제든 재사용. /skill-name 으로 호출.
